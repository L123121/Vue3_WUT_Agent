### 1.从发送问题到流式输出回答的过程

> **一句话记忆：** Fetch 获取流 → getReader 读取 → TextDecoder 解码 → 逐块更新 currentAnswer

**a. 触发与请求：** 用户输入问题，点击发送。前端将当前会话的历史消息数组构造成请求体，通过 Fetch API 发送到 Node.js 后端服务，并创建 `AbortController` 实例以支持请求取消。

**b. 流式接收：** 后端拿到请求后，调用大模型 API 时带上 `stream: true`。接着设置 SSE 响应头 `Content-Type: text/event-stream`，将大模型返回的原始 SSE 流通过管道直接转发给前端。后端只做认证和管道转发，不解析内容字段，首字节延迟最低。

**c. 前端解析与渲染：** 前端通过 Fetch API 获取响应体的 ReadableStream，使用`getReader()` 获取一个读取器（reader）。然后反复调用这个 reader 的 `read()` 方法，来逐块读取二进制字节（chunk），经 TextDecoder 增量解码后按 SSE 规范解析出内容片段。每收到一个片段，直接通过 `onChunk` 回调拼接到 currentAnswer 末尾并触发 Vue 响应式更新。触发 Vue 重新渲染时，markdown-it 将 currentAnswer 转换为 HTML，通过 v-html 插入 DOM 实时显示，实现边接收边渲染的打字机效果。

| 环节 | 你的表述 | 准确性 |
|------|----------|--------|
| 获取流 | `Fetch API 获取响应体的 ReadableStream` | ✅ 完全正确 |
| 读取 chunk | `调用 getReader() 逐块读取二进制字节获得 chunk` | ✅ **明确 chunk 来源** |
| 解码 | `经 TextDecoder 增量解码` | ✅ 完全正确 |
| 协议解析 | `按 SSE 规范解析出内容片段` | ✅ 概括性描述，技术场景下可展开 |
| 缓冲区 | `用 buffer 拼接不完整的 SSE 事件，`\n\n` 切分` | ✅ 与渲染缓冲区分离 |
| 更新 currentAnswer | `逐块 onChunk 回调直接追加，触发 Vue 响应式更新` | ✅ 完全正确 |
| 最终渲染 | `Markdown 渲染库转换为 HTML，打字机效果` | ✅ 完全正确 |

**d. 结束与存储：**
当流式传输结束或解析到 `[DONE]` 标识时，前端将用户问题与拼接完成的完整答案组合成消息对象，存入 `messages` 数组并同步更新 Pinia 状态。随后将整个会话记录序列化后写入 localStorage，完成消息的本地持久化。

**e. 异常处理：**
- 网络中断：保留已生成的部分内容，提示用户可重试
- 用户取消：通过 AbortController 中止请求
- API 报错：展示友好提示，记录错误日志

**追问：AbortController 怎么用？**

```js
const controller = new AbortController();      // ① 创建控制器

fetch('/api/chat', {
  signal: controller.signal,                    // ② 绑定 signal
});

// ③ 取消请求
controller.abort();
```

调用 `abort()` 后：
- Fetch 的 `reader.read()` 会抛 `AbortError` 异常，进入 `catch` 分支
- 服务端收到中断信号后停止生成，节省算力
- 前端保留已收到的部分内容，不触发重连

```js
try {
  const response = await fetch(url, { signal: controller.signal });
  // ... 流式读取
} catch (err) {
  if (err.name === 'AbortError') {
    // 用户主动取消，保留已有内容，不做重连
    savePartialContent(currentAnswer);
  } else {
    // 网络中断等其他异常，触发重连逻辑
    handleReconnect();
  }
}
```

**关键设计点：** `AbortError` 和网络异常要分开处理。取消是用户有意为之，不应重连；网络中断是意外，需要自动恢复。

### 2.大模型输出是字还是词？

从大模型底层的生成逻辑来说，它是基于 Token（词元） 的生成单位。Token 并不是固定的"字"或"词"，而是一种**文本分割单位**：

- 英文：一个 Token 可能是完整单词，也可能是单词的一部分
- 中文：通常一个或多个汉字作为一个 Token

**前端处理策略：**
无论底层是字还是词，前端都不需要关心具体的分割粒度，只需要：

1. 从 Chunk 中解析出 content
2. 通过 RAF 批量合并到 currentAnswer
3. Markdown 渲染库将完整内容转换为 HTML 并实时显示

```js
// 流式渲染核心：RAF 批量合并更新
let pendingContent = '';
let rafId = null;

function onChunk(content) {
  pendingContent += content;
  if (!rafId) {
    rafId = requestAnimationFrame(() => {
      currentAnswer += pendingContent;
      pendingContent = '';
      rafId = null;
    });
  }
}
```

这样就能给用户一种"流畅生成"的打字机效果，无论底层 Token 粒度如何，用户感知到的都是连续的流式响应。

### 4.怎么确定大模型的回答结束？

> **一句话记忆：** 收到 `[DONE]` → 关闭状态 → 保存消息 → 清理变量

**我使用的是标准的流式接口。** 当大模型生成结束时，API 会发送结束标志：`data: [DONE]`。

**前端处理逻辑：** 在解析 SSE 数据流的过程中，我会逐行判断接收到的数据。如果某行数据去掉 `data:` 前缀后等于 `'[DONE]'`，就说明流式响应已经完整结束。

**收到结束标志后，我会做三件事：**
1. **关闭流式拼接状态**：将 `isStreaming` 设为 `false`，隐藏"停止生成"按钮
2. **保存完整对话**：将 currentAnswer 存入 messages 数组，并持久化到 Pinia 和 localStorage
3. **清理临时变量**：清空 currentAnswer 和缓冲区，准备下一次对话

### 5.大模型的上下文记忆和 Token 限制怎么管理？

> **一句话记忆：** 携带历史记录保记忆，滑动窗口+摘要压缩防超Token

**一、上下文记忆的实现**

大模型本身是无状态的，每次请求只会看到你发的内容。为了让它能"记住"之前的对话，前端每次请求会把这个会话的所有历史消息按时间顺序构造为 `messages` 数组，连同当前问题一起发给后端：

```js
const messages = [
  { role: 'system', content: systemPrompt },
  ...conversationHistory.map(msg => ({
    role: msg.role, content: msg.content
  })),
  { role: 'user', content: currentQuestion }
]
```

这样大模型看到完整的对话历史，就能理解上下文进行连贯回复。

**二、Token 超限的处理（滑动窗口 + 摘要压缩）**

大模型有固定的上下文窗口限制（如 4K / 8K / 128K tokens），当历史对话太长时会超出限制。我的三层策略：

1. **滑动窗口**：发送前估算总 token 数，接近上限时保留最近的 N 轮对话，丢弃最早的部分
2. **摘要压缩**：如果对话极长，丢弃早期信息会导致"遗忘"。引入一个总结流程——将最早的对话压缩成一段摘要，作为 system prompt 的一部分保留关键信息
3. **语义检索（进阶方案）**：将历史关键信息向量化存储，每次查询时用 Embedding 做语义召回，把最相关的片段注入上下文。Token 利用率更高，但需要向量数据库支持

**追问：Token 预估怎么算？**
中文约 1.5-2 tokens/字，英文约 1 token/word。实际项目中可以用 `tiktoken` 等库精确计算，前端做粗略估算即可。

**追问：摘要压缩和滑动窗口各有什么优缺点？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **滑动窗口** | 实现简单，零额外开销 | 丢弃的信息永久丢失 |
| **摘要压缩** | 保留关键信息的语义 | 额外调用模型，有成本 |
| **语义检索** | Token 利用率最高，不丢失关键信息 | 需要向量数据库 + Embedding 服务

### 6.说说Copilot与Agent的区别

核心区别在于**主动性和决策能力**。

- **Copilot（副驾驶）：** 以**用户为中心**。它的作用是辅助、建议、补全。用户需要明确下达指令（如写邮件、写代码），Copilot 提供内容建议，但最终决策权和执行路径由用户主导。比如 GitHub Copilot，你写代码它补全。
- **Agent（智能体）：** 以**目标为中心**。你给它一个最终目标（如"帮我订一家餐厅"），它会自主规划步骤（打开浏览器 → 搜索餐厅 → 比较评价 → 选择一家 → 填写预定信息），并调用外部工具（Tool）来执行，直到完成任务。Agent 具有更强的自主推理和行动能力。

**追问：你的项目里 AI 是 Copilot 还是 Agent？**
我的项目定位是 **Copilot 模式**。用户每次提问，AI 基于上下文和知识库给出回答，决策权始终在用户手中。但如果引入 Function Calling（工具调用），就可以升级为轻量 Agent——比如让 AI 自动查询知识库、调用计算器、搜索网页等，实现"给目标 → 自动执行"的模式。

**追问：Agent 的工具调用（Function Calling）怎么实现？**

1. **定义工具**：后端服务调用模型 API 时，在请求体的 tools 字段中声明可用工具的名称、描述和参数 schema
2. **模型决策**：大模型根据用户意图，决定是否调用工具以及传什么参数
3. **后端执行**：后端收到模型返回的工具调用指令后，执行对应函数（如查数据库、调 API）
4. **结果回传**：将工具执行结果再次发给模型，模型基于结果生成最终回答
5. **循环判断**：模型可决定是否需要再次调用工具，直到满足用户需求

```javascript
// 后端定义工具 schema
const tools = [
  {
    type: "function",
    function: {
      name: "search_knowledge",
      description: "从知识库中搜索相关内容",
      parameters: {
        type: "object",
        properties: {
          query: { type: "string", description: "搜索关键词" },
          top_k: { type: "integer", default: 3 }
        },
        required: ["query"]
      }
    }
  }
];

// 发送请求时带上 tools
const response = await sparkAPI.chat({ messages, tools });

// 模型返回 tool_calls 后，后端执行工具
if (response.tool_calls) {
  const result = await executeTool(response.tool_calls[0]);
  // 将结果回传给模型生成最终回答
}
```

> **注意：** 工具定义在后端，只能调用后端预先定义好的工具，这是安全特性，防止 AI 乱调用。

### 7.Pinia 与 localStorage 的关联和区别

> **一句话记忆：** Pinia 管内存（快但刷新丢），localStorage 管持久化（慢但永久存），通过手动防抖同步

- **关联**：通常结合使用。Pinia 是**内存中的状态管理**，刷新页面会丢失。为了持久化，在会话变更时调用 `scheduleSaveCache()`，经 500ms 防抖后将 `conversations` 数组序列化写入 localStorage。页面加载时再从 localStorage 读取恢复。
- **条件同步**：仅当用户未登录（本地模式）时才写入 localStorage。登录用户的数据通过后端 API 持久化，localStorage 作为离线兜底：

```js
// conversation.store.js 中的同步逻辑
let saveTimer = null;
const scheduleSaveCache = () => {
  if (saveTimer) clearTimeout(saveTimer);
  saveTimer = setTimeout(() => {
    if (!isBackendAvailable()) {          // 仅本地模式写入
      saveLocalConversationsCache(conversations.value);
    }
    saveTimer = null;
  }, 500);                                // 防抖，避免频繁写入
};
```

也可以使用 `pinia-plugin-persistedstate` 插件替代手写同步——一行配置自动监听 state 变化写入 localStorage，效果等价。本项目选择手写是因为需要"仅本地模式写入"的条件判断，插件不好表达这种逻辑。

- **区别**：

- **存储位置**：Pinia 在内存（RAM），读写快，但页面刷新即失；localStorage 在硬盘，持久化存储，但读写是同步的，会阻塞 JS 主线程。数据量小时问题不大，但如果会话历史很长，几百 KB 的数据写入硬盘时，页面会明显卡顿
- **数据类型**：Pinia 可以是任何响应式对象、函数 ；localStorage 只能存储字符串（需 `JSON.stringify`）。
- **用途**：Pinia 用于组件间共享响应式数据；localStorage 用于持久化存储。

### 8.SSE 与 WebSocket 对比

> **一句话记忆：** SSE 是 HTTP 长连接单向推送，WebSocket 是全双工双向通信

**SSE (Server-Sent Events) 实现原理：** SSE 本质上是 HTTP 协议长连接。客户端通过 `EventSource` API 与服务器建立连接，服务器端保持连接打开，并持续返回特定格式（`text/event-stream`）的数据。

**追问：为什么选 SSE 不选 WebSocket？** 因为 AI 流式输出是**单向推送**场景（服务器 → 客户端），不需要双向通信。SSE 基于 HTTP，实现简单，自带断线重连，且支持 POST 请求携带完整对话历史。WebSocket 更适合聊天室、协作编辑等需要双向实时通信的场景。

**SSE两种实现方式的对比**

| 对比维度 | EventSource API（原生方式） | Fetch API（进阶方式） |
|----------|---------------------------|----------------------|
| **浏览器支持** | 除 IE 外的所有现代浏览器均原生支持，API 成熟稳定 | 所有支持 `fetch` 的现代浏览器均可使用，灵活性更高 |
| **请求方法** | **仅支持 GET 请求**，无法自定义请求头或携带请求体 | **支持任意 HTTP 方法**，可自由定义请求头和请求体 |
| **数据获取方式** | 浏览器自动处理连接、解析消息格式，通过 `onmessage` 等事件直接接收数据 | 需手动调用 `getReader()` 获取 `ReadableStream`，自行处理字节流解码和协议解析 |
| **断线重连** | **内置自动重连机制** | **需要手动实现重连逻辑** |
| **适用场景** | 无需鉴权的简单 GET 请求，希望快速实现、省心可靠的场景 | 需要 POST 请求、携带鉴权 Token、或需要对数据流进行精细自定义控制的复杂场景 |

**前端实现：**
```
// 核心代码示例（你当时需要手写的）
const eventSource = new EventSource('/api/chat/stream');

// 监听默认消息
eventSource.onmessage = (event) => {
  const data = JSON.parse(event.data);
  // 这里是渲染界面的逻辑
  updateUI(data);
};

// 监听自定义事件 (如果后端发了 event: xxx)
eventSource.addEventListener('delta', (event) => {
  console.log('Delta:', event.data);
});

eventSource.onerror = (err) => {
  console.error('SSE 连接断开或出错', err);
  // 可以在这里实现重连逻辑
};
```

- **技术难点：**
1. **连接断开与重连：** 网络波动会导致连接断开。fetch + ReadableStream 不带自动重连机制，需手动实现重连逻辑，并处理幂等性（防止消息重复）。
2. **消息格式与边界：** 需要严格按照 `data: ...\n\n` 格式解析数据。
3. **并发连接数限制：** 浏览器对同一个域名有最大 SSE 连接数限制（通常 6 个）。
4. **服务器资源管理：** 长连接会占用服务器资源，需要处理好连接池和超时机制。

- **SSE vs WebSocket vs 轮询：**
  - **短轮询：** 客户端每隔固定时间（如每5秒）发一次请求问服务器"有新数据吗？"，服务器不管有没有都立即返回。实现简单，但大部分请求都是无效的，浪费带宽，实时性也差。
  - **长轮询：** 客户端发一次请求，服务器如果没有数据就"挂起"不返回，直到有数据或超时才响应，响应后连接结束，客户端需要重新建立连接。比短轮询高效，但反复建立连接有开销。
  - **WebSocket：** 建立连接后，客户端和服务器可以**互相主动发消息**，就像打电话，双方都能随时说话。性能最好，开销最小，但协议复杂，适合游戏、实时协作等**双向**通信场景。
  - **SSE：** 建立连接后，只有服务器能主动给客户端推消息，客户端想发消息还得另外发请求。实现简单，自带断线重连和事件ID，适合**单向**数据流场景，如**LLM 流式输出、通知推送**。

**追问：ReadableStream 和 SSE 是什么关系？**

SSE 是一种**协议标准**（规定数据格式为 `data: ...\n\n`），ReadableStream 是 Fetch API 的**底层流读取接口**。现在 AI 项目普遍用 `fetch + ReadableStream` 而非原生 `EventSource`，因为 EventSource 只支持 GET，无法携带请求体和鉴权 Token。ReadableStream 支持 POST + AbortController，可手动控制读取节奏。

**追问：SSE 有哪些缺点？**

1. **单向通信**：只能服务器→客户端，客户端需要额外发请求
2. **连接数限制**：浏览器对同域名 HTTP 连接有限制（通常 6 个），SSE 会占用一个
3. **二进制数据差**：只支持 UTF-8 文本，不支持 Blob/ArrayBuffer
4. **代理兼容性**：Nginx 可能缓冲 SSE，需设置 `X-Accel-Buffering: no`
5. **无速率控制**：客户端无法"暂停"数据流

### 9.用户打开多个标签页，如何保证 Pinia 与 localStorage 的数据一致性？

这是一个典型的多标签页状态同步问题。我们采用了以下策略：

- **监听 `storage` 事件**：当某个标签页修改 localStorage 时，其他标签页会触发 `window.addEventListener('storage')`。我们在该事件中重新从 localStorage 读取数据并更新 Pinia ，从而保持多标签同步。
- **对于复杂合并**，如同时修改不同会话，简单覆盖问题不大；如果修改同一会话，我们以后端数据为准，或采用最后写入者胜出，并在 UI 上给出提示。
- **同步时机控制**：为了避免频繁写入，采用 500ms 防抖，仅本地模式时写入 localStorage，登录用户以后端 API 为准。

### 10.如果发生中断，会话保持从后端到前端怎么做

> **一句话记忆：** error 事件主检测 → 心跳兜底假死 → 指数退避重连 → 状态恢复 → 消息补发

**一、断线检测：error 事件（主要手段）**
网络断开或服务端崩溃时，Fetch 的 `reader.read()` 会抛异常，这是最直接的断线信号：

```js
try {
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    // 解析并渲染数据...
  }
} catch (err) {
  // 网络中断、服务端关闭都会走到这
  console.error('连接断开', err);
  handleReconnect();
}
```

如果是 EventSource 模式，则通过 `eventSource.onerror` 回调捕获。绝大多数断线场景（断网、服务端重启、负载均衡切换）都能被 error 事件直接捕获。

**二、心跳检测（防御性措施）**
在部署环境中，存在一种边界情况：TCP 连接看似正常，但服务端已经"死"了（进程卡死、网关超时但没有发送 TCP RST）。这时 error 事件不会触发，客户端以为连接正常，实际已经收不到数据。心跳就是 30s 发一次 `/health` 探测，连续 3 次超时判定连接失效。本项目因为是短连接 + 直连后端，出现概率很低，加心跳更多是展示防御性设计思维。

```js
let heartbeatTimer = null;
let failCount = 0;
const MAX_FAIL = 3;
const HEARTBEAT_INTERVAL = 30000;

function startHeartbeat() {
  heartbeatTimer = setInterval(async () => {
    try {
      const res = await fetch('/api/heartbeat', { method: 'HEAD' });
      if (res.ok) failCount = 0;
    } catch {
      failCount++;
      if (failCount >= MAX_FAIL) {
        clearInterval(heartbeatTimer);
        handleReconnect();  // 服务端假死，触发重连
      }
    }
  }, HEARTBEAT_INTERVAL);
}
```

**总结：** error 事件捕获正常断线，心跳兜底服务端假死，两者配合覆盖所有异常场景。

**三、断线重连流程（指数退避）**

1. **指数退避**：第1次等1秒重试，第2次等2秒，第3次等4秒...每次等待时间翻倍，上限30秒。避免频繁重试把服务器打崩。
2. **重试次数**：最多重试5次，超过则提示用户手动刷新。
3. **重连请求**：携带 `sessionId` 和 `lastEventId`，让后端知道从哪里恢复。
4. **重连成功**：恢复状态，重启心跳检测。

**四、状态恢复与消息补发**
1. **后端会话存储**：使用 Redis 存储会话上下文（对话历史、会话元数据），配合 JWT 恢复用户身份
2. **消息补发**：断线期间后端可能已生成部分回复，重连时通过 `lastEventId` 请求补发丢失的 SSE 事件
3. **前端状态恢复**：重连后从后端拉取最新会话状态，同步更新 Pinia 和 localStorage

**五、用户取消场景**
用户主动取消时，通过 `AbortController` 中止请求，保留已生成的部分内容，不触发重连逻辑。

```js
const controller = new AbortController();

// 发送请求时绑定
fetch('/api/chat', { signal: controller.signal });

// 用户点击"停止生成"
function handleStop() {
  controller.abort();
  // 保留 currentAnswer 中已有的内容
  savePartialMessage(currentAnswer);
}
```

### 11.自动滚动怎么做？在生成过程中，用户想往上查阅历史，应该怎么办？

> **一句话记忆：** 监听滚动判断用户意图，距离底部 > 阈值则暂停自动滚动，回到底部则恢复

- **自动滚动**：新内容生成时，用 `scrollTo({ behavior: 'smooth' })` 平滑滚到底部。

**追问：smooth 和 instant 滚动的区别？** `smooth` 是平滑动画过渡，用户体验好但有延迟；`instant` 是瞬间跳转，无延迟但体验生硬。流式输出场景用 `smooth` 更自然。

```
// 监听滚动，判断用户是否在查看历史
container.addEventListener('scroll', () => {
    const distanceToBottom = scrollHeight - scrollTop - clientHeight;
    autoScrollEnabled = distanceToBottom < SCROLL_THRESHOLD;
});

// 新内容生成时调用
function tryAutoScroll() {
    if (autoScrollEnabled) container.scrollTo({ top: scrollHeight, behavior: 'smooth' });
}
```

- **处理用户干预**：当用户向上滚动想看历史消息时，自动滚动会暂停，避免页面一直往下跳。用户看完后手动滚回底部，自动滚动又会恢复。

**判断用户意图的原理：**
```
距离底部的距离 = 内容总高度 - 已滚动高度 - 可视区域高度
```

| 情况 | 距离底部 | 状态 |
|------|----------|------|
| 用户在底部附近 | 小于阈值 | 开启自动滚动 |
| 用户向上滚查看历史 | 大于阈值 | 暂停自动滚动 |

distanceToBottom = scrollHeight - scrollTop - clientHeight 与阈值进行判断

| 属性 | 说明 |
|------|------|
| `scrollHeight` | 内容总高度（最顶部 → 最尾部） |
| `clientHeight` | 可视区域高度（屏幕上能看到的那一截） |
| `scrollTop` | 已滚上去的高度（最顶部 → 当前视口顶部） |

### 14.每个 chunk 都是完整 JSON 吗？

> **一句话记忆：** TextDecoder 解码 → buffer 缓冲 → `\n\n` 切分 → JSON.parse

不一定。底层 Fetch 的 chunk 是字节粒度的，由 TCP 网络包大小决定，不受应用层控制。如果直接将 chunk 当完整 JSON 解析，极大概率会遇到 JSON 结构被截断或多字节字符被从中间切开导致乱码。

正确的做法是实现一个 SSE 的**轻量级缓冲解析机制**：

1. **流式解码**：利用 `TextDecoder` 将字节流转为字符流，设置 `{ stream: true }` 让它缓存不完整的字节，等下一个 chunk 来了再一起解码，防止 UTF-8 多字节字符被截断乱码。
2. **边界缓冲**：将解码后的字符串追加到协议缓冲区 `buffer` 中，**只追加不解析**。
3. **协议切分**：按 SSE 规范，用 `\n\n` 对缓冲区进行切割。切割后，**前面的片段是完整事件**，可以推入渲染缓冲区；**最后一个片段后面没有结束符**，需要**保留回协议缓冲区**等待下一块数据拼接。
4. **数据提取**：只有切出完整事件后，才提取 `data:` 字段并进行 `JSON.parse`。

这样才能在**不丢包、不粘包、不乱码**的前提下，稳健地处理流式 JSON。

| 步骤 | 触发时机 | 依赖 |
|------|----------|------|
| ① 读取 chunk | 网络数据到达 | 无 |
| ② 增量解码 | chunk 读取后立即执行 | ① 的输出 |
| ③ 追加到 buffer + 切分 | 解码后立即执行 | ② 的输出 |
| ④ 提取 data + JSON.parse | 切出完整事件后立即执行 | ③ 切出的完整片段 |
| ⑤ 推入渲染缓冲区 | JSON 解析后立即执行 | ④ 的输出 |
| ⑥ RAF 批量更新 | 下一帧渲染前 | ⑤ 收集的多个片段 |

### 15.AI 流式输出会不会发生丢包之类的现象？

1. **TCP 层**：绝对保证可靠交付，**不丢**。但拥塞控制的重传机制会导致应用层**卡顿**，用户体感像丢包。
2. **协议层 (SSE)**：依赖 `Last-Event-ID` 做断线重传。
3. **应用层**：chunk 是按字节截断的，可能导致两个问题：
   - **UTF-8 解码问题**：chunk 可能把多字节字符（如中文，UTF-8 需要 3 字节）从中间切开。如果不设置 `TextDecoder` 的 `stream: true`，不完整的字节会被解码成乱码
   - **协议缓冲区问题**：解码后的字符串可能把一个 SSE 事件从中间截断，导致 JSON 不完整。需要用协议缓冲区拼接多个解码后的字符串，等检测到 `\n\n` 后再解析
   两个问题都会导致 `JSON.parse` 失败，表现为消息丢失。
4. **优化手段**：

- **断线重连与续传**：服务端发送时带 `id` 字段，前端重连时浏览器自动发送 `Last-Event-ID` 头，后端根据此 ID 补发丢失数据。
- **前端数据处理**：对 chunk 做缓冲与完整性检查，确保按预期分割（如按 `\n\n` 分割）。
- **异常重试**：在 `reader.read()` 的 `try/catch` 中捕获错误，实现指数退避重连策略。

### 16.流式 Markdown 渲染”乱跳”

**1.问题本质**
Markdown 解析器是**状态机**，当遇到开启但未闭合的语法符号时，会持续将后续内容当作该语法块的一部分处理，直到看见闭合为止。流式输出时，闭合符可能在几帧之后才到达，中间这段时间解析器会产生错误的结构判定。比如 AI 输出 `**bold text` 但还没发 `**`，解析器会把后续内容都渲染成加粗。

**2.核心解法：基于 md.parse() Token 流的栈匹配补全**

不用手写字符扫描器，而是**复用 markdown-it 自己的解析能力**来检测未闭合语法：

**第一步：用 md.parse() 获取 Token 流**  
调用 `md.parse(str, {})` 得到整篇文本的完整 Token 数组（block 和 inline 级别的 tokens 都包含在 `token.children` 中）。markdown-it 的 parser 已经识别了所有语法模式，不需要自己逐字符扫描。

**第二步：遍历 nesting 提取未闭合栈**  
遍历 Token 数组，遇到 `nesting === 1`（开标签）压入栈，遇到 `nesting === -1`（闭标签）从栈中弹出匹配项。

```js
const processToken = (token) => {
  if (token.nesting === 1) {
    stack.push({ type: token.type, markup: token.markup });
  } else if (token.nesting === -1) {
    const expected = token.type.replace(/_close$/, '');
    // 从栈顶向下搜索匹配——流式场景下 token 顺序可能不完全规整
    for (let i = stack.length - 1; i >= 0; i--) {
      if (stack[i].type === expected) { stack.splice(i, 1); break; }
    }
  }
};
```

**第三步：反向补全 + 整体渲染**  
遍历结束后栈里剩下的就是所有未闭合的语法。按反向顺序生成闭合标记，直接拼接到原始文本末尾，然后整体丢给 `md.render()`。补全的闭标记本身不产生可见内容（ `**` 只是关掉粗体），所以**不需要额外删 DOM 节点**。

```js
const completions = [];
for (let i = stack.length - 1; i >= 0; i--) {
  completions.push(item.markup || '\n');
}
return completions.length > 0 ? str + completions.join('') : str;
```

**关键容错：从栈顶向下搜索**  
遇到 `_close` 时不是简单 `pop()`，而是从栈顶向下遍历找匹配的 `_open`。这是因为流式场景下 token 顺序可能不完全规整——例如 `*italic **bold text*`，斜体先开启、加粗后开启，但斜体闭合法先到达。如果直接 `pop()` 会弹出加粗，补全错误。从顶向下搜索保证无论什么顺序都能正确匹配。

**代码围栏的特殊处理**  
Fence token（`` ``` ``）是单 token（`nesting: 0`），没有开闭配对。单独通过正则匹配 `^```/gm` 统计出现次数，奇数则说明有未闭合的围栏，追加 `\n` 将其闭合。

**效果对比**

| 维度 | 手写字符扫描器 | md.parse Token 流 |
|------|--------------|-------------------|
| 语法覆盖 | 6 种（手写维护） | **全部** markdown 语法（自动覆盖） |
| 代码量 | ~80 行 | ~20 行 |
| 容错性 | 中等 | 好（token 级容错 + 从顶向下搜索） |
| 渲染方式 | 需手动删补全节点 | 直接拼字符串，无需 DOM 操作 |
| 维护成本 | 高（每加语法都要改） | 低（markdown-it 升级自动受益） |

### 17.代码生成的安全性问题

核心目标是**把代码当作文本展示，绝不作为指令执行**。我按渲染管线的四个阶段层层防御：

**1. 输入预处理（去注释、标准化语言名）**
AI 原始输出进入渲染管线前，先做两步预处理：
- **去注释**：过滤掉 HTML 注释（`<!-- ... -->`），防止恶意内容藏在注释中绕过后续检测。markdown-it 的 `html: false` 虽然不会渲染注释内容，但注释中可能隐藏着配合其他漏洞的 payload，提前剔除是最干净的做法
- **语言名标准化**：将代码块语言标识映射为标准名（`js`→`javascript`、`py`→`python`、`ts`→`typescript`、`c++`→`cpp`），确保 highlight.js 能以正确的语法规则高亮。未识别的语言名会导致高亮降级为纯文本展示，不影响安全但影响体验

**2. markdown-it 渲染（`html: false` + highlight.js 高亮）**
预处理后的文本交给 markdown-it 解析为 HTML：
- **`html: false`**：这是最根本的防线。AI 输出的任何原始 HTML 标签（`<script>`、`<iframe>`、`<img onerror>` 等）在 markdown-it 解析阶段直接被当作普通文本处理，不生成对应 DOM 节点。从语法解析层面阻断标签注入，不依赖字符串替换或正则黑名单
- **highlight.js 高亮**：对代码块进行语法着色。highlight.js 只做分词和 CSS 类名标注，不生成 `innerHTML`、不执行脚本，本身不引入 XSS 风险

**3. DOMPurify 净化（白名单 + 协议过滤）**
markdown-it 输出的 HTML 再经过 DOMPurify 二次净化作为兜底：
- **标签白名单**：只允许 `div`、`span`、`pre`、`code`、`a`、`strong`、`em`、`table` 等渲染必需的标签，其余一律剔除
- **属性白名单**：只保留 `class`、`style`、`href`、`target`、`rel` 等安全属性，`onclick`、`onerror` 等事件处理器天然被拦在白名单外
- **协议过滤**：对 `href`、`src` 属性强制过滤 `javascript:`、`data:`、`vbscript:` 等危险协议。即使 markdown-it 合法生成了一个 `<a href="javascript:alert(1)">` 标签，DOMPurify 也会将其 href 改写为无害值

**4. 写入 DOM**
净化后的 HTML 通过 Vue 的 `v-html` 指令插入消息气泡。由于前三个阶段已经层层过滤，此时写入 DOM 的 HTML 已无安全风险。

**追问：已经有 `html: false` 了，为什么还要 DOMPurify？**
两者防御的场景不同：`html: false` 防的是 AI **故意输出的原始 HTML**——比如模型被诱导生成了 `<script>alert(1)</script>`。但 markdown-it 自身合法产生的 HTML（比如 `[链接](javascript:void(0))` 会生成 `<a>` 标签，或者表格/图片等）仍然可能带有恶意属性。DOMPurify 作为最后一层，专门过滤 markdown-it 合法生成的产物中不安全的属性和标签。

### 18.RAG 知识库是怎么实现的？

> **一句话记忆：** 文档上传 → 文本分割 → Embedding 向量化 → 语义检索 → 注入上下文

**一、RAG 完整流程**

1. **文档上传：** 支持 PDF、Word、TXT 等多格式文档上传
2. **文本分割：** 将长文档按语义切分为合适大小的片段（chunk）
3. **向量化：** 通过 Embedding 模型将文本片段转化为向量
4. **存储：** 向量存入数据库，建立语义索引
5. **检索：** 用户提问时，将问题向量化，语义检索 Top-K 相关片段
6. **生成：** 将检索到的片段注入提示词，LLM 基于上下文生成回答

**追问：为什么用向量检索而不是关键词搜索？** 关键词搜索只能匹配字面相似，语义检索能理解含义。比如搜"微积分考试"，能匹配到"高等数学期末试卷"，即使没有"微积分"这个词。

**二、本项目具体实现：讯飞星火知识库**

我使用的是**讯飞星火知识库**（全托管 RAG 服务），它内置了文档解析、文本分割、Embedding 向量化和语义检索能力，不需要自己搭建向量数据库。

**工作流程：**
```
上传文档 → 讯飞自动解析 + 分割 + 向量化 + 存储
                                    ↓
用户提问 → 讯飞自动向量化 → 语义检索 Top-K → 返回相关片段
                                    ↓
                            注入 Prompt → LLM 生成回答
```

**核心 API：**
```python
# 创建知识库
kb = create_knowledge_base(name="高数复习资料")

# 上传文档（讯飞自动解析+向量化）
resp = requests.post(url, headers=headers, files={"file": f})

# 检索相关片段
resp = requests.post(url, headers=headers, 
                     json={"query": query, "top_k": top_k})
results = resp.json()

# 注入 Prompt 生成回答
context = "\n".join([r["content"] for r in results["data"]])
messages = [
    {"role": "system", "content": f"基于以下资料回答：\n{context}"},
    {"role": "user", "content": user_question}
]
response = call_spark_llm(messages)
```

**追问：为什么选讯飞星火知识库，不自己搭建向量数据库？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **讯飞星火知识库** | 全托管、开箱即用、不维护基础设施 | 厂商锁定、有调用成本 |
| **ChromaDB（自建）** | 完全控制、本地部署、零成本 | 需自行搭建、配置 Embedding 模型 |
| **Milvus/Pinecone** | 高性能、适合大规模生产 | 运维成本高、不适合小项目 |

选讯飞星火的原因：校本知识库场景数据量不大，追求快速落地。全托管服务省去了搭建和维护向量数据库的工作，只需要调 API 上传和检索。

**追问：多用户知识库怎么隔离？**
讯飞支持创建多个独立知识库，每个科目创建独立知识库（通过 `kb_id` 区分），天然实现数据隔离。公共资料使用单独的知识库，所有用户可检索。

**追问：知识库怎么管理？**
支持按科目/年份/资料类型分类，上传（PDF/Word/TXT）、删除、检索文档。文档更新后重新上传，讯飞会自动重新索引。通过版本号 + MD5 哈希判断文档是否变化，未变化则跳过，变化则全量重建该文档的向量索引。

**追问：回答准确性怎么保证？**
1. **检索质量**：优化文本分割策略，确保片段语义完整
2. **来源引用**：回答中用 `[^1]` 标注引用来源，用户可点击验证
3. **Prompt 约束**：要求 LLM 只基于资料回答，降低 temperature（0.1-0.3）减少"创造性发挥"
4. **置信度评估**：检索到的片段相似度低于阈值时提示"回答可能不够准确"

**追问：RAG 检索效果怎么调优？**

| 策略 | 说明 |
|------|------|
| **Chunk 大小** | 200-500 字最合适，递归分割确保语义完整 |
| **Overlap** | 相邻 chunk 保持 10%-20% 重叠，避免边界信息丢失 |
| **混合检索** | 向量检索（语义）+ BM25（关键词）加权融合 |
| **重排序** | 初步召回 Top-20，用精细模型（Cross-Encoder）重排取 Top-5 |

> 注：以上是通用 RAG 调优方法。本项目使用讯飞星火知识库（全托管），这些策略已内置。面试时作为扩展知识展示技术深度。

### 19.流式输出与消息列表的渲染性能怎么优化？

> **一句话记忆：** 逐块更新 + 虚拟滚动 + 组件按需加载 + Markdown 缓存 + v-memo 跳过重渲染

**1.直接追加更新**
流式输出时，每个 chunk 通过 `onChunk` 回调直接拼接到 currentAnswer 末尾，触发 Vue 响应式更新。markdown-it 在 render 阶段重新解析完整的 currentAnswer 并生成 HTML。这种方式实现简单，对于千字以内的消息体感流畅。如果后续 token 到达速率极高，可以加一层 50ms 时间分片缓冲合并更新。

**2.虚拟滚动**
当对话积累数百条消息时，全部渲染 DOM 节点会导致严重的性能问题。虚拟滚动的核心思想是：**只渲染可视区域内的消息**，滚动时动态替换内容。

- 计算可视区域高度和滚动位置
- 根据每条消息的预估高度，计算当前可见的消息范围
- 只渲染可见范围 ± 缓冲区（如上下各 3 条）的消息
- 使用 `padding-top` / `padding-bottom` 撑起容器总高度

**3.重型组件异步加载**
知识库文件上传、设置面板、Prompt 面板等非首屏组件通过 `defineAsyncComponent` 延迟加载，首屏不加载这些组件的代码，用户打开对应面板时才请求。

**4.Markdown 渲染缓存**
已渲染完成的消息用 Map 缓存 `{ content → renderedHTML }`，相同内容直接复用上次渲染结果，避免重复解析。缓存上限 50 条，超过时自动淘汰最老条目。

**5.消息组件跳过重渲染**
流式输出时只有当前正在生成的消息需要高频更新。在消息循环上使用 `v-memo` 绑定 `[msg.text, msg.isError, msg.sources, msg.id === currentStreamingId]`，未变化的消息跳过整个组件的虚拟 DOM diff。


### 20.语音输入是怎么实现的？（对应简历：Web Speech API 语音输入）

> **一句话记忆：** Web Speech API → SpeechRecognition → 实时转写 → 填入输入框

**实现流程：**

1. **初始化识别器**：调用 `window.SpeechRecognition || window.webkitSpeechRecognition` 创建实例（Chrome 专属 API）
2. **配置参数**：设置 `lang: 'zh-CN'` 中文识别，开启 `continuous` 连续识别和 `interimResults` 中途结果
3. **绑定事件**：`onstart` 置 recognizing 为 true，`onresult` 遍历 `event.results` 提取 `isFinal` 的识别文本填入输入框，`onerror` 区分"无权限""无语音""其他错误"分别提示，`onend` 重置 recognizing 为 false
4. **状态反馈**：通过 `recognizing` 标志控制 UI 变化——录音中显示波浪动画，空闲显示麦克风图标
5. **提交文本**：识别结束后将转写结果填入输入框，用户确认后发送

```js
// VoiceRecorder.vue 核心逻辑（简化）
const SR = window.SpeechRecognition || window.webkitSpeechRecognition
recognition = new SR()
recognition.lang = 'zh-CN'
recognition.continuous = true
recognition.interimResults = true

recognition.onresult = (event) => {
  let final = '';
  for (let i = event.resultIndex; i < event.results.length; i++) {
    if (event.results[i].isFinal) final += event.results[i][0].transcript;
  }
  if (final) emit('transcript', final);  // 填入输入框
}

recognition.onerror = (event) => {
  if (event.error === 'no-speech') return  // 静音不算错误
  if (event.error === 'not-allowed') { emit('error', '未授予麦克风权限'); return; }
  emit('error', '语音识别出错')
}
```

**追问：为什么用 Web Speech API，不用第三方语音识别服务（如讯飞语音、百度语音）？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| **Web Speech API** | 浏览器原生支持，零额外依赖，无需申请 API Key | Chrome 独占，不支持 Safari/Firefox；精度一般 |
| **Azure 语音 SDK** | 跨浏览器（含 Edge），精度高，思维模型与 Web Speech API 一致（`recognizeOnceAsync` ↔ `onresult`） | 需 Azure 订阅 + API Key，有网络延迟 |
| **讯飞/百度语音** | 精度更高，支持多语言，跨浏览器 | 需要 API Key、网络请求、付费，有延迟 |

选用 Web Speech API 的原因：本项目定位为校园辅助工具，Chrome 是目标用户的常用浏览器。Web Speech API 零成本、零延迟、本地识别，适合实时语音转写。如果后续需要跨浏览器或更高精度，可以替换为 Azure 语音 SDK——架构上语音输入模块已封装为 `VoiceRecorder.vue`，替换实现不需要改调用方代码，因为两种方案都遵循"实时转写 → 填入输入框"的思维模型。

**追问：浏览器兼容性和安全要求怎么处理？**

1. **安全环境**：Web Speech API 要求 `https` 或 `localhost`，否则启动时报错。代码中通过 `window.isSecureContext` 检测，非安全环境提示用户切换
2. **浏览器检测**：初始化时判断 `SpeechRecognition` 是否存在，不存在时隐藏语音按钮，退化为纯文本输入
3. **降级策略**：不支持语音时不影响正常聊天功能，用户仍可通过键盘输入
4. **Edge 兼容方案**：Edge 不支持 `SpeechRecognition`，但可接入 **Azure 语音 SDK** 实现相同功能——思维模型完全一致，仅底层 API 不同。检测到 Edge 时自动切换为 Azure SDK，用户无感知

### 21.路由懒加载和首屏优化是怎么做的？（对应简历：首屏加载时间优化至 1.6s）

> **一句话记忆：** 动态 import 路由 + Vite 分包 + 按需加载组件 + 首屏加载进 1.6s

**1. 路由懒加载**

所有页面组件通过动态 `import()` 加载，只有当用户访问对应路由时才请求代码：

```js
// router/index.js
const routes = [
  { path: '/login',  component: () => import('../views/Login.vue') },
  { path: '/chat',   component: () => import('../views/AIChat.vue') },
  { path: '/knowledge', component: () => import('../views/KnowledgeBase.vue') },
]
```

这样首屏只加载 Login.vue 或 AIChat.vue 的代码，其他页面（如 KnowledgeBase）按需加载。

**2. 静态资源分包**

Vite 构建时将第三方库（Vue、Pinia、markdown-it、highlight.js 等）打包到独立的 vendor chunk 中，利用浏览器缓存长期不变的部分，用户切换页面时只需加载业务代码的增量。

```js
// vite.config.js 中的分包配置
build: {
  rollupOptions: {
    output: {
      manualChunks: {
        vendor: ['vue', 'vue-router', 'pinia'],
        editor: ['markdown-it', 'highlight.js'],
      }
    }
  }
}
```

**3. 组件异步加载**

重型组件（如知识库文件上传、设置面板、Prompt 面板）使用 `defineAsyncComponent` 延迟加载，不阻塞主渲染流程。

**优化效果：** 首屏加载时间控制在 1.6s 以内（基于 Lighthouse 测试，Fast 3G 网络）。

**追问：为什么用动态 `import()` 路由拆分，不用一次性加载所有路由？**

| 方案 | 首屏体积 | 页面切换速度 | 适用场景 |
|------|---------|-------------|---------|
| **一次性加载** | 全部 JS（约 200KB+） | 切换无延迟 | 小项目（< 5 页面） |
| **动态 import 拆分** | 仅首屏 JS（约 80KB） | 切换时按需加载（通常 < 100ms） | 多页面项目 |

本项目的聊天页 / 知识库页 / 登录页相互独立，用户通常只用一个页面。动态拆分后首屏体积降低约 60%，加载速度明显提升。切换页面的加载耗时在 100ms 以内（Vite 构建的产物小且缓存友好），用户无感知。

### 22.多会话管理是怎么实现的？（对应简历：多会话管理 + 消息持久化 + 对话上下文恢复）

> **一句话记忆：** Pinia 统一管理会话列表 + localStorage 持久化 + API 同步后端 + 删除时中止流式

**1. 会话数据结构**

每个会话包含 id、title、messages、createdAt、updatedAt 等字段，存储在 `conversation.store.js` 的 `conversations` 数组中。

**2. 会话 CRUD**

| 操作 | 实现方式 |
|------|---------|
| **创建** | `createConversation()` → 生成唯一 ID → 插入列表头部 → 设为当前会话 |
| **切换** | `switchConversation(id)` → 更新 `currentConversationId` → 加载对应消息 |
| **重命名** | `renameConversation(id, title)` → 更新 title 字段 → 同步后端 |
| **删除** | `deleteConversation(id)` → 先中止流式请求 → 移除会话 → 切换到其他会话 |

**3. 排序策略**

`sortedConversations` 按 `updatedAt` 降序排列，最新活跃的会话排在最前面。新消息到达时自动更新 `updatedAt` 触发重排。

**4. 持久化机制**

- **内存层**：Pinia 管理全部会话列表和当前激活 ID，响应式驱动 UI
- **缓存层**：localStorage 保存会话列表和最近消息，页面刷新后快速恢复
- **服务端层**：后端 API 持久化完整会话数据（登录用户），`localStorage` 作为离线缓存

**5. 流式删除保护（关键设计）**

删除正在流式输出的会话时，先用 `AbortController` 中止进行中的 fetch 请求，确保流式回调不会在会话删除后继续写入数据导致状态错乱：

```js
// chat.store.js 中的安全删除
const deleteConversation = async (id) => {
  // 如果正在流式且删除的就是当前会话，先中止请求
  if (messageStore.isLoading && currentConversationId.value === id) {
    messageStore.abortStream()  // 调用 AbortController.abort()
  }
  // 再从列表中移除
  conversations.value = conversations.value.filter(c => c.id !== id)
  // 如果删除的是当前会话，切换到其他会话或创建新会话
  if (currentConversationId.value === id) {
    currentConversationId.value = conversations.value[0]?.id || ''
    if (!currentConversationId.value) createConversation('默认会话')
  }
}
```

**追问：为什么用 Pinia 管理会话，不用 localStorage 直接管理？**

localStorage 是同步读写，每次操作会阻塞主线程。如果用 localStorage 直接管理会话列表，频繁增删切换时页面会有明显卡顿。Pinia 在内存中操作，响应式驱动 UI 实时更新，localStorage 只作为持久化层异步写入。两者的关系是：**Pinia 管交互流畅，localStorage 管数据不丢**。

**追问：删除正在流式的会话怎么处理？**

直播式请求的回调是异步的，删除后如果回调继续执行，数据会写入已删除的会话甚至污染其他会话。我的处理分三步：

1. **删除前**：用 `AbortController` 立即中止 fetch 请求，阻止后续回调执行
2. **删除中**：从 `conversations` 数组中移除该会话，切换 `currentConversationId` 到其他会话
3. **回调保护**：流式回调函数中判断当前 `activeStreamingConversationId` 是否仍是该会话，如果不是则丢弃数据
